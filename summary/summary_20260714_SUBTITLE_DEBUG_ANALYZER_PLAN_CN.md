# 字幕调试分析器开发计划

日期：2026-07-14

## 结论

该功能可以开发，现有字幕流水线已经具备实现所需的主要数据基础：16 kHz 单声道 WAV、最终字幕、原始识别字幕、VAD 分段字幕和被过滤字幕。建议将它实现为仅在“生成中间调试文件”开启时出现的独立 Tkinter 调试窗口，不影响普通用户路径。

## 目标界面

- “字幕生成”页开启 debug 后，在复选框右侧显示“调试字幕”按钮；关闭 debug 时隐藏。
- 点击按钮打开独立、可缩放的 `Toplevel` 完整窗口。
- 上半区为字幕列表，至少显示序号、开始时间、结束时间、时长、字幕文本；支持滚动与当前项高亮。
- 下半区为宽大的水平波形区，包含振幅波形、时间刻度、字幕时间块、播放游标和当前字幕文本。
- 提供打开调试目录/选择 WAV 与 SRT、播放/暂停、停止、跳转上一条/下一条、适配全长、缩放与水平滚动等控制。
- 点击字幕列表条目后，波形视口定位到对应时间，播放游标移动到字幕起点，并高亮对应字幕块。
- 播放时游标按实际播放时间前进，列表和底部字幕跟随当前时间自动切换。

## 数据与文件约定

debug 开启时，每个视频继续使用现有 `<视频名>_debug` 目录，建议形成：

- `<视频名>.wav`：保留的 16 kHz 单声道分析音频。
- `<视频名>.srt`：最终字幕的副本或明确指向相邻的 `.jp.srt`。
- `<视频名>.raw.srt`：后处理前识别结果。
- `<视频名>.vad.srt`：VAD/ASR 分块。
- `<视频名>.removed.srt`：被过滤字幕及原因。

当前 `SubtitleGenerator.transcribe()` 无条件删除临时 `.asr.wav`，因此需要调整音频生命周期：非 debug 模式保持删除；debug 模式将 WAV 移动或复制到 debug 目录，并在转录结束后保留。应避免同时在视频目录遗留 `.asr.wav` 和 debug WAV。

## 技术方案

### 1. GUI 集成

- 在 `tool_subtitle/gui.py` 中监听 `self.gen_debug_files` 的变化，动态显示/隐藏按钮。
- 新建独立模块 `tool_subtitle/debug_analyzer.py`，避免继续扩大现有 GUI 文件。
- 调试窗口不持有识别模型，只读取已生成的 WAV/SRT，关闭窗口时停止播放并释放定时器、子进程和数组。
- 若当前生成目录中存在多个调试任务，先展示可选任务列表；若仅有一个则直接载入。

### 2. 字幕读取与轨道

- 实现容错 SRT 解析器，读取时优先使用 `utf-8-sig`，兼容 UTF-8 BOM、CRLF、多行字幕和空文本。
- 默认轨道显示最终字幕；可切换 final/raw/VAD/removed，以不同颜色显示各类区域。
- 字幕块按 `start/end` 映射到波形 X 坐标，极短字幕设置最小可点击宽度。
- 点击列表、字幕块或时间轴时统一走同一个 `seek(time_sec)` 状态入口，避免多套定位逻辑不一致。

### 3. 波形生成

- 使用现有 NumPy 与 `soundfile` 读取 WAV，不新增绘图库依赖。
- 采用分桶 peak envelope：每个屏幕像素保存该时间窗的最小值和最大值，使人声形成明显波峰、静音接近水平线。
- 用 Tkinter `Canvas` 绘制背景、中心线、波形、时间刻度、字幕矩形和播放游标。
- 长音频不直接绘制全部采样点；按视口宽度降采样，并缓存多分辨率峰值数据，保证数小时视频仍可缩放和滚动。
- 波形预处理放到后台线程，所有 Tk 控件更新通过主线程 `after()` 执行。

### 4. 播放与进度同步

- 不使用 `ffplay`，也不要求安装包增加外部播放器。
- Windows 版使用 Python `ctypes` 调用系统自带 WinMM `waveOut` API，直接播放 debug WAV 的 PCM 数据；该方案不需要新增 Python 包或额外 EXE/DLL。
- 新建小型播放后端（建议 `tool_subtitle/audio_player.py`），负责设备打开、分块缓冲、播放、暂停、继续、停止、seek、当前位置查询和资源释放。
- WAV 已固定为 16 kHz 单声道，播放层仍应读取实际采样率、声道数和位深；若不是 WinMM 可直接播放的 PCM 格式，则通过现有 `soundfile`/NumPy 在内存中转换为 16-bit PCM。
- 采用多个短缓冲区循环提交，避免将数小时音频整体复制进内存；seek 时执行 `waveOutReset`、清空旧缓冲并从目标采样位置重新排队。
- 播放进度优先通过 `waveOutGetPosition` 获取实际设备播放位置，而不是只依靠墙钟估算；GUI 每 30–50 ms 更新游标和当前字幕。
- 关闭窗口、切换文件或播放结束时调用 reset/close，并保证所有 `WAVEHDR` 缓冲在释放前已经完成或取消，避免 native 崩溃。
- 后端接口与 GUI 解耦，未来如需支持非 Windows 平台，可以增加其他实现而不改分析器交互层。

### 5. 国际化与打包

- 在 `i18n/zh.json`、`i18n/en.json`、`i18n/ja.json` 增加按钮、列名、播放器状态、错误和轨道名称。
- WinMM 来自 Windows 系统组件，不需要修改安装包来携带播放器；打包验证重点是 `ctypes` 播放后端在 PyInstaller 环境下正常运行。
- JSON 修改和验证均按 UTF-8 BOM 兼容方式处理。

## 分阶段实施

1. 调整 debug WAV 保存规则，补充路径解析与 SRT 解析单元测试。
2. 完成独立窗口骨架、任务选择和字幕列表。
3. 完成波形峰值缓存、时间轴、字幕块、缩放和滚动。
4. 完成列表/波形双向定位以及当前字幕跟随。
5. 接入 WinMM `waveOut` 播放后端，完成播放、暂停、停止、seek 和游标同步。
6. 补齐中英日文案、打包项、异常提示和资源释放。
7. 使用短音频、长视频、静音片段、重叠字幕、中文路径和缺失调试文件进行回归测试。

## 验收标准

- debug 关闭时不显示按钮，也不保留 WAV；开启时按钮可见且每个完成任务保留可分析 WAV。
- 调试窗口能加载最终字幕及三类现有 debug SRT，字幕列表与波形时间一致。
- 人声与静音在波形上有明显差异，长音频操作无明显卡顿。
- 点击任一字幕条目后，视口、字幕块、播放游标和底部字幕同时定位正确。
- 播放、暂停、停止和再次 seek 后，游标误差肉眼不可察觉；关闭窗口不残留播放器进程。
- 普通字幕生成流程、停止流程、模型清理和非 debug 临时文件清理不发生回归。

## 风险与建议

- 最大风险转为 WinMM 缓冲区生命周期管理：`WAVEHDR` 和 PCM 缓冲必须在系统播放完成前保持有效引用，并在关闭设备前正确 unprepare/reset。
- WinMM 方案针对当前 Windows 产品；如果未来支持 Linux/macOS，需要另行提供平台播放后端。
- 超长视频的波形必须采用分桶/缓存方案，不能按原始采样逐点画 Canvas。
- 生成任务未完成时 WAV 可能仍在写入，首版建议只加载已完成任务，避免边生成边读取的复杂度。
- 首版以“分析和定位”为范围，不加入字幕文本编辑、拖拽改时间和保存回写；这些适合作为第二阶段编辑器能力。
